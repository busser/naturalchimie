# Sprite Metadata and Authoring Tool

This file specifies how sprite art is integrated into the game.
It covers two related deliverables:

1. The **metadata model** — how sprite PNGs and a small JSON file
   describe a sprite well enough that the renderer can place it
   correctly in a grid cell.
2. The **authoring tool** — a small standalone HTML application
   that lets a human author the metadata visually, with live
   preview, instead of eyeballing pixel coordinates in an image
   editor.

The metadata model is what the runtime game depends on. The
authoring tool is a development-time aid; it does not ship with
the game.

## Why this exists

Sprite art for this game is generated by an AI image model. AI-
generated sprites do not come out at predictable pixel
dimensions. The wheat sprite has fanned tips that extend beyond
the bundle's main body. The apple's main body might be 240px wide
and 220px tall, not perfectly square. The detonator is taller
than other sprites because of its plunger. Forcing every sprite
into a fixed canvas would require either fighting the model or
manually trimming/padding each output, both error-prone.

The metadata model lets each sprite be whatever shape it
naturally is, and stores per-sprite numbers describing how to
align and scale it into a grid cell. The cost of this flexibility
is that someone has to author those numbers. The authoring tool
makes that fast.

## Metadata model

Each sprite consists of two artifacts:

- A PNG image with a transparent background, of any dimensions.
- An entry in a shared `sprites.json` file describing how to
  place that PNG into the grid.

The JSON entry has exactly the following fields:

```json
{
  "tier-07-apple": {
    "file": "tier-07-apple.png",
    "anchor": [120, 230],
    "cell_width_px": 240,
    "cell_height_px": 220
  }
}
```

Field semantics:

- **`file`** — path to the PNG, relative to the JSON file.
- **`anchor`** — `[x, y]` pixel coordinates within the source
  image, identifying the bottom-center of the sprite's
  cell-footprint. The anchor is the point that the renderer aligns
  to the cell's bottom-center on screen.
- **`cell_width_px`** — width in source pixels of the rectangular
  region of the source image that should be mapped to one cell
  width on screen.
- **`cell_height_px`** — height in source pixels of that
  rectangular region.

The **cell-footprint rectangle** is implicitly defined by these
three values:

- Left edge: `anchor.x − cell_width_px / 2`
- Right edge: `anchor.x + cell_width_px / 2`
- Bottom edge: `anchor.y` (i.e., the anchor sits on the bottom
  edge)
- Top edge: `anchor.y − cell_height_px`

This rectangle is the source-image region that maps exactly onto
one grid cell on screen. Anything in the source PNG outside that
rectangle (the potion's neck and cork, the apple's stem and leaf,
the wheat's fanned tips) renders **outside** the cell on screen,
scaled by the same factors. Extruding parts above the rectangle
appear in the cell above; nothing should extrude below or to the
sides in practice (the spec calls for upward extrusion only).

## Render math

To draw a sprite into a cell:

1. Look up the sprite's metadata (`anchor`, `cell_width_px`,
   `cell_height_px`).
2. Compute the source rectangle:
   ```
   src_left   = anchor.x − cell_width_px / 2
   src_top    = anchor.y − cell_height_px
   src_right  = anchor.x + cell_width_px / 2
   src_bottom = anchor.y
   ```
3. Compute the horizontal and vertical scale factors:
   ```
   scale_x = cell_size_px / cell_width_px
   scale_y = cell_size_px / cell_height_px
   ```
   where `cell_size_px` is the on-screen size of one grid cell
   (a renderer setting; the same value for every sprite).
4. Draw the **entire source image** (not just the source
   rectangle) onto the screen, scaled by `scale_x` horizontally
   and `scale_y` vertically, positioned so that the source image's
   anchor pixel lands at the cell's on-screen bottom-center.

Step 4 is the subtle one: the renderer does not crop the source
image to the cell-footprint rectangle. It draws the whole source
image, but uses the rectangle to define the scale and the anchor
point. This is what allows extruding parts (potion necks, apple
stems, etc.) to render outside the cell naturally, scaled
consistently with the main body.

### Implementation pseudocode

```typescript
interface SpriteMetadata {
  anchor: [number, number];  // [x, y] in source pixels
  cell_width_px: number;
  cell_height_px: number;
}

interface SpriteAsset extends SpriteMetadata {
  image: HTMLImageElement;   // or equivalent for the renderer
}

function drawSpriteAtCell(
  ctx: CanvasRenderingContext2D,
  sprite: SpriteAsset,
  cell_screen_x: number,    // left edge of cell on screen
  cell_screen_y: number,    // top edge of cell on screen
  cell_size_px: number,     // size of one cell on screen
): void {
  const scale_x = cell_size_px / sprite.cell_width_px;
  const scale_y = cell_size_px / sprite.cell_height_px;

  // Where the anchor lands on screen: cell's bottom-center.
  const anchor_screen_x = cell_screen_x + cell_size_px / 2;
  const anchor_screen_y = cell_screen_y + cell_size_px;

  // Where to place the source image's top-left on screen,
  // such that source.anchor maps to anchor_screen.
  const draw_x = anchor_screen_x - sprite.anchor[0] * scale_x;
  const draw_y = anchor_screen_y - sprite.anchor[1] * scale_y;

  ctx.drawImage(
    sprite.image,
    draw_x,
    draw_y,
    sprite.image.width * scale_x,
    sprite.image.height * scale_y,
  );
}
```

This function is the **single source of truth** for sprite
rendering. The game's render loop calls it for every visible cell.
The authoring tool calls it for its preview. If the two produce
different output, that's a bug — most likely a mismatch in
which version of the function each is using. **Extract this
function into a shared module** (e.g., `src/sprite-renderer.ts`)
that both the game and the authoring tool import from.

## Z-ordering

Per `04-visual-style.md`, lower-row sprites render in front of
higher-row sprites, so that upward-extruding parts (potion necks,
apple stems) are not occluded by the sprites in the cells above.

To implement: sort visible cells by row ascending (row 1, the
floor, drawn last), then draw each in order. Within the same row,
column order does not matter — sprites in the same row do not
overlap horizontally.

```typescript
function renderBoard(...) {
  // Lower rows render later (i.e., on top).
  const cells_in_draw_order = sort_cells_by_row_descending(board);
  // Wait — descending so row 7 is drawn first, row 1 last.
  for (const cell of cells_in_draw_order) {
    drawSpriteAtCell(...);
  }
}
```

For animations involving moving sprites (drop, gravity), the
"row" used for z-sorting is the sprite's **current visual row**,
which may be a fractional value mid-animation. A sprite at row
3.7 (falling from row 4 to row 3) draws in front of any sprite at
visual row 4 or higher.

## Authoring guidance

When authoring metadata for a new sprite, the procedure is:

1. **Identify the main body.** Look at the sprite and decide which
   part should fit inside one cell. For a potion, that's the bulb
   (excluding neck and cork). For an apple, the apple itself
   (excluding stem and leaf). For a wheat bundle, the central
   bundle (excluding fanned tips). For sprites with no extruding
   parts (eyeball, bone, crystal, ore, gold), the entire sprite is
   the main body.

2. **Place the anchor.** It is at the bottom-center of that main
   body. For a potion, dead center horizontally, at the very
   bottom of the bulb. For an apple, dead center horizontally, at
   the bottom of the apple's body.

3. **Measure `cell_width_px`.** It is the width of the main body's
   bounding box in source pixels. The main body's left edge is at
   `anchor.x − cell_width_px / 2`; right edge at
   `anchor.x + cell_width_px / 2`. Eyeball it; precision to within
   a few pixels is enough.

4. **Measure `cell_height_px`.** It is the height of the main
   body's bounding box, measured upward from the anchor.

For most sprites, `cell_width_px` and `cell_height_px` will be
nearly equal (the main body is roughly square). The two values
exist separately to allow small non-uniform scaling for sprites
whose main body isn't perfectly square — for example, an apple
whose body is 240×220 px should render filling the cell, which
requires a slightly larger vertical scale factor than horizontal.

### Constraint on non-uniform scaling

The horizontal and vertical scale factors should not differ by
more than ~15%. Beyond that, sprites look distorted (a stretched
apple looks wrong). If you find a sprite whose `cell_width_px` and
`cell_height_px` differ by more than ~15%, the right fix is to
either:

- Accept that the sprite is intentionally smaller in one
  dimension than the cell (set both values to match the larger
  dimension; the sprite will sit a bit small in its cell); or
- Regenerate the sprite to have a more square main body.

## File layout

The expected layout in the project:

```
public/
  sprites/
    sprites.json
    tier-01-green-potion.png
    tier-02-yellow-potion.png
    ...
    tier-12-gold-nugget.png
    special-dynamite.png
    special-detonator.png
```

The `public/` directory is Vite's convention for static assets that
should be served as-is at the site root, so a sprite at
`public/sprites/tier-01-green-potion.png` is reachable in the browser
at the URL `/sprites/tier-01-green-potion.png`.

`sprites.json` keys correspond to logical sprite names. The game
maps tiers to sprite names via a small lookup table — this layer
of indirection means the asset filenames can change without
touching gameplay code.

Sprite naming convention:

- Elements: `tier-NN-<name>.png` where `NN` is the zero-padded
  tier number (01–12) and `<name>` is a single-word lowercase
  identifier matching the spec (e.g., `green-potion`, `chocapic`,
  `gold-nugget`).
- Special items: `special-dynamite.png`, `special-detonator.png`.

## The authoring tool

The authoring tool is a single-page HTML application that loads
the sprites and the JSON file, lets the human edit metadata
visually, and writes updates back to disk.

### Goals

- Make per-sprite metadata fast to author and tweak.
- Show the sprite the way the game will render it, as the human
  edits the metadata, with no save / reload roundtrip.
- Surface the cell-footprint rectangle visually so the human
  knows exactly what they're describing.
- Ship as a single file with no build step and no server, so it
  is trivial to maintain and to run on any machine.

### Tech stack

A single self-contained HTML file (`sprite-tool.html`) using:

- Vanilla HTML, CSS, and JavaScript (no framework).
- Two `<canvas>` elements: one for the source image with overlays,
  one for the preview.
- Form inputs for direct numeric editing of the metadata fields.
- The browser's File System Access API to read and write
  `sprites.json` and to list PNG files in the folder. Fall back to
  drag-and-drop and copy-to-clipboard if the API is unavailable.

The tool depends on the same `drawSpriteAtCell` function as the
game (see "Render math" above). The function lives in a shared
module that the tool's HTML file imports — for example, by
copying it inline as a `<script>` block, or by referencing a
sibling JavaScript file. The exact import mechanism is up to the
implementer, but the **function used by the tool and the function
used by the game must be the same code**, not two copies.

### Sprite list

A vertical list of all sprites loaded from the folder. Clicking
one selects it; the source-image panel and preview update to
show that sprite.

The list shows each sprite by its name (the JSON key, e.g.,
`tier-07-apple`). A small visual indicator next to each name
shows whether its metadata has been authored yet — for example, a
filled dot for "metadata exists" and an empty dot for "no
metadata, defaults will be used."

### Source-image panel

The selected sprite's PNG is displayed at its native resolution,
on a checkered background suggesting transparency.

Three overlays are drawn on top:

- **Image bounds** — a thin border around the entire PNG, so the
  human sees what's inside vs. outside the source image.
- **Anchor crosshair** — a small high-contrast crosshair (e.g.,
  red plus sign with a white outline) at the anchor pixel.
- **Cell-footprint rectangle** — a translucent box of dimensions
  `cell_width_px × cell_height_px`, positioned so that its
  bottom-center coincides with the anchor. The box has a
  semi-opaque colored fill (e.g., translucent yellow) and a clear
  border. This rectangle is the visualization of "what part of
  the source image will fit inside one cell on screen."

The overlays are interactive:

- **Click on the source image** sets the anchor to the clicked
  pixel. The crosshair moves immediately.
- **Drag the corners of the cell-footprint rectangle** resizes it,
  updating `cell_width_px` and `cell_height_px`. The rectangle
  remains anchored at the anchor point, so dragging the
  bottom-left or bottom-right corner moves only that corner;
  dragging the top corners adjusts the height.

A toggle in the toolbar lets the user hide/show each overlay
independently for cases where they obscure the sprite.

### Metadata fields

Below the source image, four numeric input fields:

- `anchor.x` (integer, 0 to image width).
- `anchor.y` (integer, 0 to image height).
- `cell_width_px` (integer, 1 to image width).
- `cell_height_px` (integer, 1 to image height).

Each field is bidirectionally bound to the visual overlay: typing
in a field updates the overlay; manipulating the overlay updates
the field. Changes from either source persist immediately.

### Preview panel

A canvas at the bottom of the right column, showing a 3×3 grid
with the currently-selected sprite drawn in every cell using
`drawSpriteAtCell`. The cells are drawn at a representative
on-screen size (e.g., 64×64 pixels — the same size the game uses,
or some chosen authoring size) with faint cell boundary lines so
the human can see whether the sprite fills its cell correctly.

The preview shows immediately what the sprite will look like
in-game when surrounded by other sprites. Updates live as
metadata changes.

#### Two-sprite preview mode

A toggle switches the preview from "3×3 of this sprite" to "this
sprite below another sprite." When enabled, the user picks a
second sprite from a dropdown, and the preview shows a 3×3 grid
with the selected sprite in the bottom row and the dropdown
sprite in the top two rows. This is useful for verifying z-order:
that an apple's stem properly extends into the cell above, and
that the potion in the cell above doesn't visually clash with the
extruding stem.

### Persistence

Auto-save on every metadata change. The tool writes
`sprites.json` to disk via the File System Access API.

If the API is unavailable (e.g., on Firefox at the time of
writing), the tool falls back to:

- Showing the current full JSON in a read-only text area at the
  bottom of the page.
- Providing a "Copy JSON" button that copies the current
  `sprites.json` content to the clipboard.

Auto-save eliminates the Cmd+S habit and keeps the JSON file in
sync with the visual state. There is no undo stack inside the
tool; users rely on `git diff` and `git checkout` for undo, which
is fine for an authoring tool used by developers.

### Loading

On open, the tool prompts the user to select the sprite folder
(via the File System Access API's directory picker, or via a
drag-and-drop fallback for browsers without the API).

Once loaded, the tool:

1. Lists all PNG files in the folder.
2. Loads `sprites.json` if present; otherwise creates an empty
   one.
3. For each PNG with a metadata entry, populates that sprite's
   fields from the JSON.
4. For each PNG without a metadata entry, populates default
   metadata (`anchor` at the geometric center of the bottom edge
   of the image; `cell_width_px` and `cell_height_px` equal to
   the image's dimensions). This default is rarely correct but
   is a usable starting point.

### Out of scope for v1

- Editing the source PNG images (use an image editor for that).
- Generating sprites (use the AI image tooling for that, see the
  prompts in the conversation history).
- Rendering the full game preview with running gameplay (the
  game itself is the right tool for that).
- Undo/redo within the tool.
- Multi-sprite operations (e.g., "set `cell_width_px` for all
  selected sprites at once").
- Validation warnings (e.g., "scale factors differ by more than
  15%"). Nice-to-have but not essential.

These can be added in later versions if they prove useful in
practice.

### Implementation notes for Claude Code

When building the tool:

- Keep it in a single HTML file.
- Use `<canvas>` for both the source-image panel and the preview;
  do not try to position DOM overlays on the source image (HTML
  positioning will fight you on retina displays and zoom).
- Use the same `drawSpriteAtCell` function defined in this
  document for the preview rendering. If the game's renderer is
  also being built in the same project, share the function via a
  module import. If the tool is being built first as a
  prototype, paste the function inline and plan to refactor.
- Use `requestAnimationFrame` to redraw the source-image canvas
  and the preview canvas whenever any metadata value changes. No
  need for a fancy diff or virtual DOM; redrawing two small
  canvases on every change is well within the performance budget.
- The File System Access API is straightforward but requires the
  page to be loaded over `https://` or `localhost`, not opened
  directly via `file://`. A short note in the tool's UI should
  warn the user if they've opened it via `file://` and suggest
  serving it via `python3 -m http.server` or similar.
- Save metadata changes by writing a JSON.stringify of the in-
  memory metadata dictionary to `sprites.json`, with two-space
  indentation. The file should be human-readable; developers will
  occasionally want to inspect or edit it manually.

### Acceptance criteria

The tool is "done" when:

- It loads a sprite folder and `sprites.json` from disk.
- The sprite list reflects all PNGs in the folder, with each
  sprite's selection state visually clear.
- Selecting a sprite shows its source image with the three
  overlays.
- Clicking the source image moves the anchor; dragging the cell-
  footprint rectangle's corners resizes it.
- All four metadata fields are bidirectionally bound to the
  visuals.
- The 3×3 preview updates live as metadata changes.
- The two-sprite preview mode works.
- Metadata changes persist to `sprites.json` on disk
  automatically (or to the clipboard via a button as fallback).
- Both the tool and the game's render loop use the same
  `drawSpriteAtCell` function.

## Summary of integration with the rest of the spec

This file specifies the **glue** between sprite art (which is
generated externally, per the prompts in conversation history)
and the runtime renderer (whose visual rules are in
`04-visual-style.md` and timings are in `05-animations.md`). The
contract is small:

- Each sprite is a PNG plus a JSON entry with anchor, width, and
  height.
- The renderer scales the source image so that the anchor lands
  at the cell's bottom-center on screen and the cell-footprint
  rectangle maps exactly to the cell's on-screen size.
- Z-order draws lower rows in front, ensuring extruding parts of
  upper sprites don't occlude lower neighbors.

If the runtime ever produces unexpected sprite positioning, the
debugging path is:

1. Verify the metadata in `sprites.json` against what the
   authoring tool shows.
2. Verify both tool and game are calling the same
   `drawSpriteAtCell` function.
3. If both check out, the bug is in the function itself, and
   fixing it fixes both tool and game simultaneously.
